Replit Agent 4 Instruction — Automatic Job Closure Email to Unsuccessful Applicants
Read this entire instruction before starting. Implement each sprint in order. Do not move to the next sprint until the current one is tested and working.

Context
When a job is closed on EventLink — either because the employer manually closes it, or because the event date has passed — all applicants who were not hired should automatically receive a polite notification email. The email thanks them for applying, explains the role has been filled, and nudges them to strengthen their EventLink profile using the Build My Reputation tool.
This is an automated trigger, not a manual send. It fires once per job, when the job status changes to closed or completed.

Run the migration before proceeding.

status is NOT hired (or whatever the hired status value is in your schema — check the applications table status enum)
closureEmailSent is false

Sprint 3 — Trigger: Fire on job close
There are two places this function must be called. Find both in the codebase:
Trigger A — Employer manually closes a job
Find the route or controller function that handles PATCH /api/jobs/:id or equivalent where the job status is set to closed, filled, completed, or inactive. Look in job.controller.ts or job.route.ts.
After the job status is successfully updated to closed/filled, call:
typescriptimport { sendJobClosureEmails } from "../services/job-closure-email.service";

// After successful status update:
sendJobClosureEmails(jobId).catch(err =>
  console.error(`Job closure email error for job ${jobId}:`, err)
);
Use .catch() so that an email failure never crashes the job close response. The employer should not see an error even if the emails fail.
Trigger B — Scheduled job for expired events (event date has passed)
Look for the existing scheduler in server/api/services/job-notification-scheduler.service.ts (built in a previous sprint). Add a new cron job that runs daily at 10:00am UK time:
typescript// Daily at 10:00am UK time — close expired jobs and send closure emails
cron.schedule('0 10 * * *', async () => {
  const now = new Date();

  // Find all active jobs where eventDate < now (event has passed)
  const expiredJobs = await db
    .select()
    .from(jobs)
    .where(
      and(
        // Status is still active/open — not already closed
        not(eq(jobs.status, 'closed')),
        not(eq(jobs.status, 'filled')),
        not(eq(jobs.status, 'completed')),
        // Event date has passed
        lt(jobs.eventDate, now)
      )
    );

  for (const job of expiredJobs) {
    // Mark job as closed
    await db
      .update(jobs)
      .set({ status: 'closed', updatedAt: now })
      .where(eq(jobs.id, job.id));

    // Send closure emails to unsuccessful applicants
    await sendJobClosureEmails(job.id).catch(err =>
      console.error(`Scheduled closure email error for job ${job.id}:`, err)
    );
  }

  if (expiredJobs.length > 0) {
    console.log(`Daily scheduler: closed ${expiredJobs.length} expired jobs and sent closure emails`);
  }
}, { timezone: 'Europe/London' });
Important: Check what the actual status field values are in the jobs table before writing these conditions. Use the exact string values from the schema, not assumptions.

Sprint 4 — Guard: Prevent duplicate sends
The closureEmailSent flag on each application handles per-applicant deduplication. But also add a check at the top of sendJobClosureEmails to prevent the function being called twice for the same job in rapid succession (e.g. if a webhook fires twice):
typescript// At the top of sendJobClosureEmails, before any DB queries:
const alreadySentCount = await db
  .select({ count: sql`count(*)` })
  .from(applications)
  .where(
    and(
      eq(applications.jobId, jobId),
      eq(applications.closureEmailSent, true)
    )
  );

// If any closure emails have already gone for this job, skip entirely
if (Number(alreadySentCount[0].count) > 0) {
  console.log(`Job ${jobId} closure emails already sent — skipping`);
  return;
}

Sprint 5 — Admin visibility
In the admin dashboard (wherever jobs are listed in the admin view), add a small indicator showing how many closure emails were sent for each job. This can be a simple count query:

Display this as a badge next to the job in the admin job list — e.g. "✉ 12 notified". This lets Loris confirm at a glance that closure emails went out correctly for each job.
